# ADR-007: Use UUID v7 as Entity Identifiers

## Status

**Accepted**

## Context

The domain requires entity identifiers that are independent from the persistence layer.

Using database-generated auto-increment identifiers couples entity creation to the database and makes future distributed architectures more difficult to implement.

UUID version 7 combines global uniqueness with chronological ordering, providing better database index locality than UUID version 4 while preserving identifier independence.

## Decision

All aggregate roots and entities use UUID version 7 generated by the domain instead of auto-increment integer identifiers generated by the database.

Identifier generation is considered a domain responsibility rather than a persistence concern.

## Consequences

### Positive

- Domain entities remain persistence independent.
- Identifiers can be generated before persistence.
- Better index locality than UUID v4.
- Globally unique identifiers.
- Facilitates future distributed and event-driven architectures.

### Negative

- UUIDs require more storage than integer identifiers.
- UUIDs are less human-readable than sequential integers.
- UUID support must be available throughout the application.
